# v6.0.0b — Intelligence Layer: Cognee Sync & PDF Ingestion

## Context

Soleri's CogneeClient is fire-and-forget — if Cognee/Docker is down, ingestion silently fails. Meanwhile, there's no way to ingest reference books into the vault. These two features complete the intelligence layer.

## Feature 1: Smart Cognee Sync (#125)

### Problem

Three vault write paths (seed, update, remove) don't track Cognee sync state. If Cognee is offline, data is lost. No dirty tracking, no retry, no visibility.

### Design

A `CogneeSyncManager` class backed by a `cognee_sync_queue` SQLite table intercepts vault writes, queues sync ops, and drains to Cognee when available.

```
Vault.seed/update/remove → SyncManager.enqueue(op) → cognee_sync_queue
                                                            │
                                                      drain() ← startup / timer / health-flip
                                                            │
                                                  CogneeClient.addEntries / delete
                                                            │
                                                    mark cognee_ingested_hash on success
```

### Schema

New table:

```sql
CREATE TABLE IF NOT EXISTS cognee_sync_queue (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  op TEXT NOT NULL,              -- 'ingest' | 'update' | 'delete'
  entry_id TEXT NOT NULL,
  dataset TEXT NOT NULL,
  content_hash TEXT,
  status TEXT NOT NULL DEFAULT 'pending',
  attempts INTEGER NOT NULL DEFAULT 0,
  error TEXT,
  created_at INTEGER NOT NULL,
  processed_at INTEGER
);
```

New column on entries table: `cognee_ingested_hash TEXT` via ALTER TABLE.

### Drain Logic

- **Startup reconciliation:** Scan entries where `content_hash != cognee_ingested_hash`, enqueue dirty ones.
- **Health-flip detection:** When CogneeClient transitions unavailable → available, trigger drain.
- **Batch processing:** 10 ops per drain cycle, max 3 retries per op, then mark failed.
- **Graceful degradation:** If Cognee stays offline, queue grows but nothing blocks.

### Vault Integration

Vault gets `setSyncManager(mgr)` method. When set, `seed()`, `update()`, `remove()` auto-enqueue. Optional — if not wired, vault works exactly as before.

### New Ops (3)

| Op | Auth | Description |
|----|------|-------------|
| `cognee_sync_status` | read | Queue stats: pending/processing/failed counts, last drain |
| `cognee_sync_drain` | write | Force immediate drain attempt |
| `cognee_sync_reconcile` | write | Full reconciliation scan + enqueue dirty entries |

## Feature 2: Book/PDF Ingestion (#126)

### Problem

No way to ingest reference books into the vault. Salvador has a full 6-stage intake pipeline; Soleri needs the same capability.

### Design

An `IntakePipeline` class with SQLite-backed job tracking and 6-stage processing. Uses `pdf-parse` for text extraction and `LLMClient` for GPT classification.

### Pipeline Stages

1. **Initialize** — Parse PDF, extract TOC from first pages via LLM, create chunks
2. **Chunk** — Break into logical units (chapters from TOC or fixed N-page chunks)
3. **Classify** — LLM classifies each chunk into knowledge types (pattern, anti-pattern, principle, concept, reference, workflow, idea, roadmap)
4. **Dedup** — TF-IDF cosine similarity check against existing vault entries (threshold: 0.85)
5. **Store** — Persist classified items to vault with citation metadata
6. **Finalize** — Update job status, compute stats

### Schema

```sql
CREATE TABLE IF NOT EXISTS intake_jobs (
  id TEXT PRIMARY KEY,
  status TEXT NOT NULL,           -- initialized | processing | completed | failed | cancelled
  config TEXT NOT NULL,           -- JSON: pdfPath, title, author, chunkPageSize, etc.
  pdf_meta TEXT,                  -- JSON: totalPages, fileHash, fileSize
  toc TEXT,                       -- JSON: extracted TOC entries
  stats TEXT,                     -- JSON: items extracted/stored/deduped/failed, cost
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  completed_at INTEGER
);

CREATE TABLE IF NOT EXISTS intake_chunks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  job_id TEXT NOT NULL REFERENCES intake_jobs(id),
  chunk_index INTEGER NOT NULL,
  title TEXT,
  page_start INTEGER NOT NULL,
  page_end INTEGER NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  items_extracted INTEGER DEFAULT 0,
  items_stored INTEGER DEFAULT 0,
  items_deduped INTEGER DEFAULT 0,
  error TEXT,
  processed_at INTEGER
);
```

### LLM Integration

Uses `runtime.llmClient.call()` for classification. Benefits from existing circuit breaker, retry logic, and key pool. Classification prompt returns structured JSON: each chunk produces 0-N items with type, title, description, tags, severity, citation.

### Dedup Gate

Reuses Curator's existing TF-IDF cosine similarity infrastructure. Items with >0.85 similarity to existing vault entries are flagged as duplicates and skipped.

### New Dependency

`pdf-parse` in `packages/core/package.json` — lightweight (~200KB), pure JS PDF text extraction.

### New Ops (4)

| Op | Auth | Description |
|----|------|-------------|
| `intake_ingest_book` | write | Initialize intake job from PDF path |
| `intake_process` | write | Process next N chunks of a job |
| `intake_status` | read | Job progress or list all jobs |
| `intake_preview` | read | Preview classification on page range without storing |

### Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Manifest storage | SQLite tables | Consistent with engine, queryable |
| PDF parser | pdf-parse | Lightweight, pure JS, popular |
| LLM integration | LLMClient directly | Reuses circuit breaker, retry, key pool |
| Cognee sync | Queue + drain | Resilient to offline, no data loss |
| Dedup | TF-IDF cosine via Curator | Reuses existing infrastructure |
