# Soleri v7: File-Tree Architecture Shift

**Date:** 2026-03-16
**Status:** Approved
**Plan ID:** plan-BUILD-flow-1773644747679
**Breaking:** Yes — no backward compatibility

## Motivation

Building AI agents as coded TypeScript frameworks is operating at the wrong abstraction layer. Every AI agent maps to a simple file tree:

- **Workflows** = Folders
- **Tasks** = Subfolders
- **Each task** contains: Instructions + Tools + Data

Claude Code already reads files, follows markdown instructions, and calls MCP tools. The agent definition should be a folder tree that the model reads natively — not generated TypeScript that duplicates what the model does.

Soleri's **knowledge engine** (vault, brain, curator, learning, memory) is the real product — it solves a problem no model update will replace: persistent, cross-session, cross-project learning. But the engine is currently tangled with routing/scaffolding plumbing that belongs in the file tree.

### What This Changes

| Layer | Before | After |
|-------|--------|-------|
| Agent definition | Generated TypeScript project | Folder tree (yaml, markdown, json) |
| Tool routing | Facade factory → op dispatch → handler | Direct MCP tool registration (grouped by module) |
| Scaffolding | 22 TypeScript templates → `npm install` → `tsc` | `mkdir` + copy templates → done |
| CLAUDE.md | Generated by TypeScript function | Auto-composed from `agent.yaml` + `instructions/*.md` |
| Knowledge engine | Mixed into agent code | Standalone MCP server |

### What Does NOT Change

- `@soleri/core` internals (vault, brain, curator, planner, memory, learning modules)
- SQLite schema (no data migration)
- Knowledge packs (JSON bundles work in both architectures)
- Cognee integration (engine-level, unaffected)
- Learning loop (engine-level, unaffected)

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Tool naming | Grouped: `soleri_vault op:search` | 50 flat tools is noisy; grouping aids model discoverability. But no facade factory — direct handler dispatch. |
| CLAUDE.md | Auto-generated, never manually edited | `soleri dev` watches `agent.yaml` + `instructions/` and regenerates on change |
| Backward compat | None — clean cut | Moving forward. Old TS agents are not supported. |
| Validation gate | Salvador ported first | Hardest agent proves the architecture works at scale |
| E2E tests | Must pass after every phase | Existing test suite is the quality gate |

## Agent Folder Structure

```

├── agent.yaml                  # identity: id, name, role, tone, model, domains, principles
├── CLAUDE.md                   # AUTO-GENERATED — do not edit

├── instructions/               # behavioral rules (composed into CLAUDE.md)

│   ├── output-formatting.md    # how to present results
│   ├── clean-commits.md        # commit rules

│
├── workflows/                  # playbooks as folders
│   ├── feature-dev/
│   │   ├── prompt.md           # step-by-step workflow instructions
│   │   ├── gates.yaml          # checkpoints, acceptance criteria
│   │   └── tools.yaml          # which MCP tools this workflow uses
│   ├── bug-fix/
│   │   ├── prompt.md
```

### agent.yaml Schema

```yaml
id: salvador                    # kebab-case, unique
name: Salvador                  # display name
role: Design System Intelligence Engine
description: >
  Specialized agent for design system science, color theory,
  token migration, and component architecture.
domains:
  - design
  - accessibility
  - typography
  - color
  - component-patterns
principles:
  - Vault is the single source of truth
  - Consistency over novelty
  - Accessible by default
  - Every pixel needs a reason
tone: precise                   # precise | mentor | pragmatic | eccentric
model: claude-sonnet-4-20250514
engine:
  server: soleri-engine         # MCP server name in .mcp.json
  vault: ~/.salvador/vault.db   # vault location
  learning: true                # enable brain/learning loop
```

### .mcp.json

```json
{
  "mcpServers": {
    "soleri-engine": {
      "command": "npx",
      "args": ["@soleri/engine", "--agent", "./agent.yaml"]
    }
  }
}
```

The engine reads `agent.yaml` at startup to configure which vault to connect, which domains to load, and which knowledge packs to seed.

### CLAUDE.md Auto-Generation

Composed from (in order):
1. **Engine rules** — shared across all agents (planning gates, vault-first, output formatting, clean commits, knowledge capture)
2. **Agent identity** — from `agent.yaml` (role, tone, principles, activation commands)
3. **Instructions** — concatenated from `instructions/*.md` (sorted alphabetically)
4. **Facade table** — auto-generated from engine's registered tools
5. **Workflow index** — list of available workflows from `workflows/*/`

Regenerated by `soleri dev` on any file change. Never committed to git (add to `.gitignore`).

## Architecture: Knowledge Engine

The knowledge engine is a single MCP server (`@soleri/engine`) that exposes operations grouped by module:

```
soleri_vault     op: search | capture | capture_quick | query | connect | status | ...
soleri_brain     op: recommend | strengths | feedback | stats | ...
soleri_curator   op: groom | dedup | health | contradictions | ...
soleri_plan   op: create | approve | split | reconcile | complete | ...
soleri_memory    op: search | capture | promote | session_capture | ...
soleri_loop      op: loop_start | loop_iterate | loop_status | loop_cancel | ...
soleri_admin     op: health | diagnostic | tool_list | ...
```

### Implementation: No Facade Factory

**Before (current):**
```typescript
// facade-factory.ts — routing layer on top of MCP
const ALL_FACADES: FacadeConfig[] = [vaultFacade, brainFacade, ...];
function registerFacades(server) {
  for (const facade of ALL_FACADES) {
    server.tool(facade.name, facade.schema, async (params) => {
      const op = facade.ops[params.op];  // <-- extra dispatch layer
      return op.handler(params.params);
    });
  }
}
```

**After:**
```typescript
// direct-registration.ts — one function per module, no factory
function registerVaultTools(server, runtime) {
  server.tool("soleri_vault", schema, async ({ op, params }) => {
    switch (op) {
      case "search": return runtime.vault.search(params);
      case "capture": return runtime.vault.capture(params);
      // ... direct calls to core modules
    }
  });
}

function registerBrainTools(server, runtime) {
  server.tool("soleri_brain", schema, async ({ op, params }) => {
    switch (op) {
      case "recommend": return runtime.brain.recommend(params);
      // ...
    }
  });
}
```

Key difference: no generic factory, no FacadeConfig abstraction, no handler-capture adapters. Each module registers its own tools directly. The switch is simple, explicit, and debuggable.

## Risk Register

| Risk | Impact | Mitigation |
|------|--------|------------|
| 50+ ops in grouped tools may hit MCP tool description limits | Medium | Keep descriptions concise; test with Claude Code |
| E2E tests coupled to facade factory internals | Medium | Update tests in Phase 2 before removing factory |
| Salvador port reveals missing capabilities in file-tree format | High | Salvador goes first specifically to catch this early |
| `soleri dev` file watching adds complexity | Low | Use chokidar or native fs.watch; keep it simple |
| Domain packs need new integration point | Medium | Domain packs inject into `knowledge/` as JSON bundles |

## Success Criteria

1. `soleri create my-agent` produces a folder tree with no TypeScript
2. Claude Code reads the folder, follows instructions, calls engine tools — agent works
3. Salvador running as folder-tree agent has full capability parity with current TS version
4. All E2E tests pass
5. No facade factory code remains in codebase
