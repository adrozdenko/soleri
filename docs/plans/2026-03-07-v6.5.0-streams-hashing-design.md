# v6.5.0 Closeout — Streams & Content Hashing Design

## Context

Three remaining issues in the v6.5.0 milestone add foundational primitives to `@soleri/core`:

- **#164** ReplayableStream: multi-consumer async stream replay
- **#165** Recursive normalizer: flatten nested async/sync inputs
- **#166** Content-addressable hashing: vault deduplication at capture time

## Feature 1: ReplayableStream (#164)

New module: `packages/core/src/streams/replayable-stream.ts`

```typescript
class ReplayableStream<T> implements AsyncIterable<T> {
  private buffer: T[] = [];
  private source: AsyncIterator<T>;
  private done: boolean = false;
  private waiters: Array<{ resolve: (v: IteratorResult<T>) => void }> = [];

  constructor(source: AsyncIterable<T>);
  [Symbol.asyncIterator](): AsyncIterator<T>;
  async collect(): Promise<T[]>;
  get bufferedCount(): number;
}
```

Each call to `[Symbol.asyncIterator]()` returns an independent iterator starting from buffer index 0. The source async iterable executes exactly once — subsequent consumers replay from the buffer.

## Feature 2: Generic Normalizer (#165)

New file: `packages/core/src/streams/normalize.ts`

```typescript
type NestableInput<T> =
  | T
  | T[]
  | Promise<T | T[]>
  | AsyncIterable<T>
  | Iterable<NestableInput<T>>;

async function* normalize<T>(input: NestableInput<T>): AsyncIterable<T>;
async function collect<T>(source: AsyncIterable<T>): Promise<T[]>;
```

Recursively flattens any nesting depth into a flat `AsyncIterable<T>`. Type-agnostic — works with any T (IntelligenceEntry, string, etc.). No Soleri-specific Message type.

Discrimination logic: Promise check first, then Symbol.asyncIterator, then Symbol.iterator (excluding strings), then yield as leaf value.

## Feature 3: Content-Addressable Hashing (#166)

New file: `packages/core/src/vault/content-hash.ts`

```typescript
function computeContentHash(entry: HashableEntry): string;
```

Normalizes entry fields (lowercase domain, trim strings, sort tags, sort keys), JSON-serializes, SHA-256 hashes, truncates to 40 hex chars.

**Hashed fields:** type, domain, title, description, tags, example, counterExample.
**Excluded fields:** id, severity, timestamps, metadata (mutable/non-semantic).

### Schema Change

```sql
ALTER TABLE entries ADD COLUMN content_hash TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS idx_entries_content_hash ON entries(content_hash);
```

Migration in `vault.initialize()` with backfill: compute hash for all existing entries that lack one.

### Capture Flow Change

`vault.seed()` computes content_hash before INSERT. On conflict (hash already exists), skip the entry and return the existing ID. This makes deduplication proactive rather than reactive.

### New Ops (+2)

| Op                   | Auth | Description                                 |
| -------------------- | ---- | ------------------------------------------- |
| `vault_content_hash` | read | Compute hash for an entry without inserting |
| `vault_dedup_status` | read | Report hash coverage and duplicate stats    |

## Testing Strategy

- `replayable-stream.test.ts`: single consumer, multi-consumer, error propagation, empty stream, collect()
- `normalize.test.ts`: strings, arrays, nested arrays, promises, async iterables, mixed nesting, empty inputs
- `content-hash.test.ts`: determinism, normalization (whitespace, tag order, key order), truncation length
- `vault.test.ts`: duplicate detection at seed, backfill migration, content_hash in search results
- `vault-extra-ops.test.ts`: vault_content_hash, vault_dedup_status ops

## Constraints

- Zero new npm dependencies (node:crypto for hashing)
- Streams module is pure library code — no ops, no facade wiring needed
- Content hash backfill runs once during initialize() — safe for existing vaults
- Unique index on content_hash prevents race conditions on concurrent capture
