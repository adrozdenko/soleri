# Cross-Platform Compatibility

Soleri is designed to work across AI coding tools, not just Claude Code.

## Architecture Layers

```
┌─────────────────────────────────────────────┐
│  Host Client (Claude Code, Cursor, etc.)    │
│  Reads CLAUDE.md / instructions             │
│  Connects via MCP protocol                  │
└──────────────────┬──────────────────────────┘
                   │ MCP (stdio or HTTP/SSE)
┌──────────────────▼──────────────────────────┐
│  Soleri Engine (@soleri/core)               │
│  Transport-agnostic runtime                 │
│  Vault, Brain, Planner, Curator — pure TS   │
└─────────────────────────────────────────────┘
```

## Transport Support

| Transport    | Status     | Use case                                      |
| ------------ | ---------- | --------------------------------------------- |
| **stdio**    | Production | Claude Code, Codex — default for local agents |
| **HTTP/SSE** | Production | Multi-client, remote agents, web dashboard    |

The engine runtime (`vault.ts`, `brain.ts`, `planner.ts`, etc.) has zero MCP dependency. MCP is a transport layer, not a core dependency.

## IDE Support

The scaffold generates config files for multiple hosts:

| Host        | Config file                                    | Generated by scaffold |
| ----------- | ---------------------------------------------- | --------------------- |
| Claude Code | `.mcp.json`                                    | Yes                   |
| OpenCode    | `opencode.json`                                | Yes                   |
| Codex       | Registered via `soleri install --target codex` | CLI command           |

The `soleri install` command supports `--target claude`, `--target codex`, or `--target all`.

## What's Host-Specific

| Concern                      | Host-specific?                             | Notes                                              |
| ---------------------------- | ------------------------------------------ | -------------------------------------------------- |
| MCP tool registration        | No — standard MCP protocol                 | Any MCP client works                               |
| CLAUDE.md generation         | Partially — file name is Claude convention | OpenCode reads same file via `instructions` config |
| Prompt templates             | No — engine serves via MCP prompt resource | Any client can request                             |
| Vault, brain, planner        | No — pure TypeScript, no IDE dependency    | Works headless                                     |
| File watching (`soleri dev`) | No — uses Node.js `fs.watch`               | OS-level, not IDE-specific                         |

## What Would Break Cross-Platform

- Relying on Claude Code-specific environment variables
- Using Claude Code hook system for core logic (hooks are optional enhancement)
- Assuming specific MCP client behavior beyond the spec

## Verification

The E2E test suite (`e2e/mcp-transport.test.ts`) tests both stdio and HTTP/SSE transports. The engine boots and serves tools identically regardless of transport.
